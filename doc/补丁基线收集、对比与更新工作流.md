# 补丁基线收集、对比与更新工作流

## 目标

为 QuantumMaster 和 CombatMaster 的每个配置项建立可追溯的补丁基线，确保后续游戏版本更新时可以快速回答 3 个问题：

1. 这个配置项当前是如何 patch 的。
2. 它依赖的目标游戏函数现在长什么样。
3. 新版本到来后，这个配置项是否需要更新，以及需要更新到什么程度。

当前脚本入口：

`tools/patch-baseline/PatchBaseline.ps1`

反编译游戏源码根目录的读取优先级：

1. 显式传入 `-GameSourceRoot`
2. `Directory.Build.props` 中的 `<PatchBaselineGameSourceRoot>`
3. 回退默认值 `D:\code\Data`

## 工作流范围

这套流程覆盖四段：

1. 基线收集
2. 版本对比
3. 更新处置
4. 更新后回写

不是只做“记一份函数位置”，而是同时保存：

1. 配置项级别的 patch registry
2. 对应目标函数的完整代码正文
3. 每个版本的完整反编译游戏源码快照
4. 对比报告和更新审查模板

## 默认输出位置

默认输出根目录：

`d:\code\QuantumMaster\.cache\patch-baseline`

该目录已加入 `.gitignore`，避免把大体积版本产物默认混进仓库。

如果后续需要把基线产物放到共享盘或单独仓库，可以通过 `-OutputRoot` 指定其他位置。

## 版本目录结构

基线收集产物：

```text
.cache/patch-baseline/
  versions/
    <version>/
      manifest.json
      registry.json
      registry.csv
      inventory.md
      function-snapshots/
        QuantumMaster/
        CombatMaster/
      game-source/
```

版本对比产物：

```text
.cache/patch-baseline/
  comparisons/
    <base>-to-<target>/
      comparison.json
      comparison.md
      update-review-template.json
```

## 一、基线收集

### 做什么

1. 按 `ConfigManager.cs` / `CombatConfigManager.cs` 的单个配置 Key 建立 registry entry。
2. 优先从 `QuantumMaster.cs` / `CombatMaster.cs` 的 patch 注册表中找到这个 Key 对应的直接 patch 入口。
3. 如果某个 Key 没有直接注册，但在已注册 patch class 内部被引用和消费，则把它记为 `InternalConsumption`，而不是 `Unmapped`。
4. 下钻到 patch 文件，提取：
   1. patch 类型：ClassPatch / PatchBuilder / Hybrid
  2. 绑定方式：DirectRegistration / InternalConsumption
  3. 目标原函数
  4. 替换规则和 occurrence
  5. 上下文注入方式
  6. “如何 patch / 改什么”的摘要
5. 到反编译游戏源码目录定位目标函数，抽取完整函数代码正文并单独保存。
6. 可选地完整复制一份当前版本的反编译游戏源码到版本目录。

### 常用命令

最小试跑：

```powershell
powershell -ExecutionPolicy Bypass -File .\tools\patch-baseline\PatchBaseline.ps1 `
  -Action Collect `
  -Version pilot-20260429 `
  -IncludeConfigKey collectResource,steal,QianNianZui,XinWuDingYi
```

正式收集并保留完整游戏源码：

```powershell
powershell -ExecutionPolicy Bypass -File .\tools\patch-baseline\PatchBaseline.ps1 `
  -Action Collect `
  -Version game-1.0.0-build12345 `
  -CopyGameSource
```

指定反编译源码目录和输出目录：

```powershell
powershell -ExecutionPolicy Bypass -File .\tools\patch-baseline\PatchBaseline.ps1 `
  -Action Collect `
  -Version game-1.0.0-build12345 `
  -GameSourceRoot D:\code\Data `
  -OutputRoot D:\patch-baselines `
  -CopyGameSource
```

如果已经在 `Directory.Build.props` 中配置了 `<PatchBaselineGameSourceRoot>`，通常不需要每次都传 `-GameSourceRoot`。

### 产物说明

`registry.json`

1. 机器读取的完整基线。
2. 包含每个配置项的 patch 定义、`ConfigBinding`、目标函数签名、完整函数代码正文和 hash。

`registry.csv`

1. 轻量审阅视图。
2. 便于快速筛查哪些配置项映射到了哪些 patch 类和目标函数。

`inventory.md`

1. 人工阅读清单。
2. 直接看每个配置项的 patch 方式、绑定方式和目标函数。

`function-snapshots/`

1. 每个目标函数单独一份 `.cs` 快照。
2. 不只记录路径或行号，而是记录完整函数代码正文。

`game-source/`

1. 只在 `-CopyGameSource` 打开时生成。
2. 用来保留该版本完整反编译游戏源码，确保以后能回溯当时的整体上下文。

## 二、版本对比

### 做什么

1. 读取两个版本的 `registry.json`。
2. 按配置项比较 patch 定义是否变化。
3. 按目标函数比较：
   1. 函数是否存在
   2. 签名是否变化
   3. 完整函数代码 hash 是否变化
   4. occurrence 画像是否变化
4. 输出对比报告和更新审查模板。

### 常用命令

```powershell
powershell -ExecutionPolicy Bypass -File .\tools\patch-baseline\PatchBaseline.ps1 `
  -Action Compare `
  -BaseVersion game-1.0.0-build12345 `
  -TargetVersion game-1.0.1-build12500
```

### 对比产物

`comparison.json`

1. 机器可读对比结果。
2. 按配置项记录状态、原因、推荐动作。

`comparison.md`

1. 人工审阅摘要。
2. 快速看哪些配置项 unchanged，哪些 review-needed。

`update-review-template.json`

1. 更新审查模板。
2. 初始写入 `ComparisonStatus`、`RecommendedAction`、`Reasons`。
3. 后续人工补 `ReviewState`、`FinalDecision`、`Notes`。

## 三、更新处置

推荐按以下顺序处理游戏更新：

1. 更新反编译源码目录到新版本。
2. 先对“旧 patch 代码 + 新游戏源码”跑一次 `Collect`，生成新版本基线。
3. 跑 `Compare` 对比旧版本和新版本。
4. 打开 `comparison.md` 和 `update-review-template.json`，优先处理：
   1. `patch-redesign-needed`
   2. `patch-change-needed`
   3. `doc-only`
5. 修改补丁代码后，再对同一个新版本重新跑一次 `Collect`，刷新新版本基线。
6. 如有必要，再跑一次 `Compare`，确保对比结果符合预期。

## 四、更新后回写

每次 patch 适配完成后，至少要回写这几样东西：

1. 新版本完整反编译游戏源码快照
2. 新版本目标函数代码快照
3. 新版本 `registry.json`
4. 更新审查模板中的最终决定和说明

这样后续再更新时，比较的就不是“老版本 patch 记忆”，而是上一次已经适配过的新基线。

## 推荐的审查结论

建议在 `update-review-template.json` 的 `FinalDecision` 中使用固定值：

1. `no-update-needed`
2. `doc-only`
3. `patch-change-needed`
4. `patch-redesign-needed`

## 当前限制

1. Harmony-only 的目标方法如果无法从 patch 文件中推断准确参数个数，签名会显示为 `?`。
2. occurrence 检查目前基于源码中的方法名匹配，不是 IL 级别分析，所以它是高价值信号，但不是最终裁决。
3. `InternalConsumption` 依赖 patch class 内部对配置项的静态引用分析；它适合识别像 `FixedPagePos` 这种由其他主配置项驱动的辅助开关，但仍应结合实际 patch 代码审阅。
4. `-CopyGameSource` 会复制完整反编译源码，时间和空间开销都比较大，建议为版本命名和输出目录预留足够空间。

## 验收清单

每次正式收集或版本更新后，至少检查：

1. `manifest.json` 的 `EntryCount` 是否符合预期。
2. `inventory.md` 是否能读出每个配置项是如何 patch 的。
3. `function-snapshots/` 里是否真的保存了完整函数代码正文。
4. 如果使用了 `-CopyGameSource`，`game-source/` 是否存在并可打开源码文件。
5. 没有直接注册但被 patch class 内部消费的配置项，是否被标记为 `InternalConsumption` 而不是 `Unmapped`。
6. `comparison.md` 的结论是否与实际变化一致。
7. `update-review-template.json` 是否已被人工填入最终更新决定。
